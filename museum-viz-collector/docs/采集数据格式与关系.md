# 当前采集数据格式与关系

本文档记录截至 2026-07-06 本项目已经收集到的数据格式、存储位置，以及各类数据之间的关系。后续讨论分析方法时，可以把它作为“当前数据长什么样”的基准。

## 1. 数据存储位置

当前采集系统采用一人一稿的文件式存储，不使用数据库。

- 每个采集者对应一个用户目录：`museum-viz-collector/data/submissions/user_<hash>/`
- 采集正文保存在：`museum-viz-collector/data/submissions/user_<hash>/draft.json`
- 语音输入日志保存在：`museum-viz-collector/data/submissions/user_<hash>/voice_recordings.jsonl`
- 上传文件保存在：`museum-viz-collector/data/uploads/user_<hash>/...`
- JSON 中的媒体字段通过 `url` 指向上传文件，例如 `/exhibition_uploads/user_<hash>/photos/<file>.jpg`

`user_<hash>` 由采集者姓名和手机号生成。也就是说，系统当前的持久化单位不是“多次提交记录”，而是“一个采集者身份对应一份当前稿件”。

## 2. 顶层草稿 Draft

每份 `draft.json` 表示一位采集者针对一次展览采集到的数据。

```ts
type Draft = {
  id: string;
  createdAt: string;
  updatedAt: string;
  info: SubmissionInfo;
  floorplanAssets: MediaAsset[];
  units: Unit[];
  status?: "draft" | "submitted";
  submittedAt?: string;
};
```

字段含义：

- `id`：草稿 ID。后端会把它规范成当前用户的 `user_<hash>`。
- `createdAt`：草稿创建时间。
- `updatedAt`：最近保存时间。
- `info`：采集者、博物馆、展览等基础信息。
- `floorplanAssets`：该展览层面的平面图文件列表。
- `units`：本次展览中划分出的展厅单元列表。
- `status`：后端运行态字段，通常为 `draft` 或 `submitted`。
- `submittedAt`：提交时间；只有已经提交的草稿有这个字段。

## 3. 基础信息 SubmissionInfo

`info` 描述的是采集者和一次展览任务的基本背景。

```ts
type SubmissionInfo = {
  submitterName: string;
  submitterOrg: string;
  submitterPhone: string;
  submitterEmail: string;
  visitDate: string;
  city: string;
  museumName: string;
  museumAddress: string;
  exhibitionName: string;
  exhibitionPeriod: string;
  exhibitionLink: string;
  exhibitionIntro: string;
};
```

可以把这层理解为“展览级元数据”。后续分析中，`museumName`、`exhibitionName`、`city`、`visitDate` 可以作为分组变量；`submitterName` 和 `submitterPhone` 更偏向采集身份管理，不建议直接作为内容分析维度。

## 4. 展厅单元 Unit

`units` 是展览下的空间或叙事单元。每个单元可以包含环境照片和多个可视化项。

```ts
type Unit = {
  id: string;
  serial: string;
  name: string;
  description: string;
  environmentAssets: MediaAsset[];
  items: VizItem[];
};
```

字段含义：

- `id`：单元内部 ID，用于前端编辑和语音日志关联。
- `serial`：单元序号，例如 `单元1`、`单元2`。
- `name`：单元名称。
- `description`：对展厅单元或空间环境的文字描述。
- `environmentAssets`：单元环境照片列表，通常用于记录展区整体空间、展墙、动线、展柜等。
- `items`：该单元下具体的可视化项。

分析时，`Unit` 是连接“展览整体背景”和“具体可视化项”的中间层。它适合承载空间位置、展陈叙事段落、主题区域等分析。

## 5. 可视化项 VizItem

`VizItem` 是当前采集数据中最核心的分析对象。每个可视化项属于一个展厅单元。

```ts
type VizItem = {
  id: string;
  serial: string;
  title: string;
  locationDescription: string;
  floorplanLocation: FloorplanLocation | null;
  description: VizItemDescription;
  photos: MediaAsset[];
};
```

字段含义：

- `id`：可视化项内部 ID，用于媒体、语音日志和前端编辑关联。
- `serial`：该单元内的可视化项序号。
- `title`：可视化项名称。
- `locationDescription`：文字形式的位置描述。
- `floorplanLocation`：平面图上的点位；目前只有部分数据填写。
- `description`：围绕该可视化项的结构化文字描述。
- `photos`：该可视化项的现场照片。

`VizItem` 可以展开成一张主要分析表：一行对应一个可视化项，附带它所在的采集者、博物馆、展览、单元、照片数量、描述文本等字段。

## 6. 可视化项描述 VizItemDescription

每个可视化项的描述被拆成 5 个文本字段。

```ts
type VizItemDescription = {
  visualizationSelf: string;
  exhibitionFunction: string;
  humanInteraction: string;
  evaluation: string;
  additionalInfo: string;
};
```

字段含义：

- `visualizationSelf`：可视化描述。关注图表形态、视觉编码、数据故事等。
- `exhibitionFunction`：可视化在展览中的功能。关注它在策展叙事和信息补充中的作用。
- `humanInteraction`：空间布局与交互形式。关注观众如何在真实空间中观看、移动、触摸或参与。
- `evaluation`：采集者评价。包含主观感受、现场观察、保存状态、改进建议等。
- `additionalInfo`：其他补充，非必填。

这 5 个字段是后续质性编码、主题分析、文本分类和案例比较的主要材料。前 4 个字段在当前采集流程中是必填项，`additionalInfo` 是可选项。

## 7. 媒体资产 MediaAsset

平面图、环境照片、可视化项照片、录音文件都使用同一套媒体资产结构。

```ts
type MediaAsset = {
  id: string;
  role: "environment" | "floorplan" | "photo" | "audio";
  label: string;
  name: string;
  originalName?: string;
  type: string;
  size: number;
  dataUrl?: string;
  url?: string;
  createdAt: string;
};
```

字段含义：

- `id`：媒体文件 ID，通常也是文件名主体。
- `role`：媒体角色。
- `label`：媒体标签。当前照片标签包括 `周围环境`、`完整视图`、`关键细节`、`其他`，环境照通常为 `环境照`，平面图通常为 `展览平面图`。
- `name`：后端保存后的文件名。
- `originalName`：上传前的原始文件名。
- `type`：MIME 类型，例如 `image/jpeg`、`audio/webm`。
- `size`：文件大小，单位为字节。
- `dataUrl`：本地暂存字段。保存到服务器前可能存在，写入服务器草稿时会被剥离。
- `url`：服务器访问路径，是分析时定位真实文件的主要字段。
- `createdAt`：上传时间。

媒体资产和具体对象的关系由所在数组决定：

- `draft.floorplanAssets[]`：属于展览整体。
- `unit.environmentAssets[]`：属于某个展厅单元。
- `item.photos[]`：属于某个可视化项。
- `voice_recordings.jsonl` 中的录音文件：通过 `unitId`、`itemId`、`section` 指向具体文本输入位置。

## 8. 平面图点位 FloorplanLocation

可视化项可以记录在平面图上的相对坐标。

```ts
type FloorplanLocation = {
  assetId: string;
  xRatio: number;
  yRatio: number;
};
```

字段含义：

- `assetId`：对应 `draft.floorplanAssets[]` 中某张平面图的 `id`。
- `xRatio`：横向比例坐标，范围 0 到 1。
- `yRatio`：纵向比例坐标，范围 0 到 1。

它保存的是比例坐标，不是像素坐标，因此可以适配不同显示尺寸。分析时可用它研究可视化项在展览空间中的分布，但当前填写量还比较少。

## 9. 语音输入日志 voice_recordings.jsonl

语音输入不会直接作为 `MediaAsset` 写入 `draft.json`。系统会把原始录音文件保存到 `data/uploads/user_<hash>/audio/`，并在对应用户目录追加一行 JSON 日志。

单条日志结构如下：

```ts
type VoiceRecordingLog = {
  id: string;
  file: string;
  url: string;
  createdAt: string;
  unitId?: string;
  itemId?: string;
  section?: string;
  sectionLabel?: string;
  mimeType: string;
  size: number;
  status: string;
  engine?: string;
  text: string;
};
```

字段含义：

- `id`：录音 ID。
- `file` / `url`：录音文件名和访问路径。
- `createdAt`：录音时间。
- `unitId`：关联的展厅单元。
- `itemId`：关联的可视化项。
- `section`：关联到哪个文本字段，例如 `visualizationSelf`。
- `sectionLabel`：文本字段的中文标签。
- `mimeType` / `size`：录音文件格式和大小。
- `status`：识别状态，例如 `ok`、`asr_failed`、`convert_failed`。
- `engine`：实际使用的语音识别引擎分支。
- `text`：识别出的文本。

语音日志是“输入过程数据”，不是最终稿件的唯一文本来源。最终用于分析的文本仍以 `draft.json` 里的 `description` 字段为准。
